2D 生存建造 Demo
项目简介
独立开发的 Unity 2D 生存建造原型，围绕「白天采集建造、夜晚依靠营火生存」的核心节奏搭建完整玩法闭环。

---
 技术栈
 UI ： UGUI、TextMeshPro 
 数据配置 ： ScriptableObject（道具 / 建筑配方） 
 架构模式 ：有限状态机（FSM）、事件总线（观察者 / 发布订阅）、命令模式、面向接口编程、泛型单例 
 性能 ： 对象池（双缓冲回收） 
 持久化 ： 主存档为 JSON 文本（`JsonUtility` 写入 `persistentDataPath/save_game.json`，；轻量设置用 `PlayerPrefs`

---

 核心脚本 / 模块说明

 1. 玩家状态机（FSM）

采用经典有限状态机管理玩家 Idle、Move、Harvest（采集）、Attack（近战）、Building（建造预览）、Dead 六种状态，每种状态封装独立的 `Enter / Exit / HandleInput / Update / FixedUpdate`，解决了多动作冲突问题——例如攻击硬直期间屏蔽移动输入，建造预览中由建造系统占用左键、近战入口短路。地面状态（Idle / Move）在 `HandleInput` 里按「死亡 → 采集 → 建造」优先级处理后再调用战斗组件的「尝试处理攻击输入」，避免在 `Update` 里与 FSM 抢切状态。状态切换由各状态内部判定并调用 `StateMachine.ChangeState()`，避免了一个巨型 switch 集中处理所有行为。

> 脚本：`PlayerStateMachine.cs`、`PlayerState.cs`（抽象基类）、`PlayerIdleState.cs`、`PlayerMoveState.cs`、`PlayerAttackState.cs`、`PlayerHarvestState.cs`、`PlayerBuildingState.cs`、`PlayerDeadState.cs`

 2. 事件总线与事件驱动架构

全局 `EventBus` 基于泛型发布/订阅实现模块间通信：Controller 修改 Model 后 `Publish` 结构体事件，View 层订阅刷新 UI，模块之间不直接引用。广播时对订阅列表做快照拷贝，避免回调内增删订阅导致的枚举异常。同时用 `lock` 保证字典的线程安全，为后续 Job System 回调预留了安全切入点。

旧模块通过 `GameEvents` 静态事件桥接，新模块优先走 `EventBus`，实现渐进式解耦而非一次性大重构。

> 脚本：`EventBus.cs`、`GameEvents.cs`、`GameEventPayloads.cs`

 3. 建造系统与命令模式

建造流程支持配方校验、材料消耗、网格吸附、半透明预览、碰撞检测、工作台范围限制。建造操作封装为 `PlaceBuildingCommand`（实现 `IGameCommand` 接口），通过 `CommandInvoker` 统一调度并维护撤销栈（限深 32）——扣费与生成在 `Execute` 中完成，撤销时 `Undo` 销毁实体并退还材料，天然支持后续扩展重做、操作日志或联机回放。配方 SO 通过 `isInventoryRecipe` 字段区分世界放置与背包合成两种产出路径，同一套数据结构承载建造与合成两条流程。

> 脚本：`BuildingSystem.cs`、`IGameCommand.cs`、`PlaceBuildingCommand.cs`、`CommandInvoker.cs`、`BuildingRecipeSO.cs`（ScriptableObject 配方）

 4. ScriptableObject 数据驱动

道具配置 `ItemData` 和建筑配方 `BuildingRecipeSO` 均继承 ScriptableObject，严格分离只读策划配置与运行时数据。道具 SO 存放在 `Assets/Data/Items/`，配方 SO 存放在 `Assets/Prefabs/Recipes/`，编辑器内可视化编辑、天然支持 Sprite / Prefab 等引用类型，新增道具或配方只加资产不改代码。

> 脚本：`ItemData.cs`、`BuildingRecipeSO.cs`、`ItemDatabase.cs`

 5. 对象池（双缓冲回收）

`PoolManager` 管理掉落物、敌人等高频生成对象。归还的实例先进入回收缓冲队列，在下次取出前统一合并回主池（`LateUpdate` 兜底），避免同帧内高频 Enqueue/Dequeue 导致的逻辑抖动与性能抖动。池配置通过 Inspector 序列化数组设置，支持运行时按需延迟注册。

> 脚本：`PoolManager.cs`、`PooledObject.cs`

 6. JSON 存档与迁移

采用 `JsonUtility` 将 `SaveData` 序列化为 `Application.persistentDataPath/save_game.json`（`prettyPrint` 便于调试与人工排查）。持久化字段含玩家坐标、生存数值、昼夜时间、背包（木/石/浆果三类）、装备六槽（整型枚举）、任务进度、营火坐标与燃料等；营火列表使用数组而非 `List`，以满足 JsonUtility 限制；`saveFormatVersion` 预留后续字段扩展。读档顺序：新 JSON → 旧 `save.json` → 旧 `save_game.bin`，保存成功后删除旧 bin/旧 json，只保留单一主文件；读档后清空建造撤销栈。主菜单 Continue 仍通过 `HasSaveFileOnDisk()` 判断是否存在任一可加载文件。注：一般「已放置建筑（除营火流程外）」与资源点枯竭状态当前未纳入该 JSON，与完全还原世界仍有差距。

> 脚本：`SaveManager.cs`、`SaveData.cs`、`BinarySaveIO.cs`、各子数据类

 7. 泛型单例基类

`SingletonMonoBehaviour<T>` 提供统一的 `Instance` 访问、重复实例销毁、可选 `DontDestroyOnLoad`，规范了 PoolManager、SaveManager 等全局管理器的生命周期管理，避免散落的手写单例逻辑。

> 脚本：`SingletonMonoBehaviour.cs`

 8. 昼夜循环与生存属性

`DayNightCycle` 用归一化时间驱动 白天→黄昏→夜晚 三阶段循环，阶段切换时通过 `GameEvents.RaiseDayPhaseChanged` 广播（内部同步 `EventBus.Publish(DayPhaseChangedPayload)`），敌人、UI、营火安全区、黑暗伤害等模块可订阅静态事件或总线，无需互相引用。生命/饥饿双属性在 `PlayerStats` 中管理，夜间远离营火时 `DarknessDamageController` 持续扣血，与营火安全区半径判定联动。

> 脚本：`DayNightCycle.cs`、`PlayerStats.cs`、`DarknessDamageController.cs`、`Campfire.cs`

 9. 敌人 AI

敌人在夜间激活，进入侦测范围后向玩家移动，到达停止距离时停步播放攻击动画（`Attack` Trigger），接触伤害由独立碰撞体（`Is Trigger`）的 `OnTriggerStay2D` 结算。攻击动画期间通过 `IsInAttackState()` 检查 AnimatorStateInfo 禁止位移，实现「先停再攻」的手感。实体碰撞体（非 Trigger）保证玩家不会穿透敌人。

> 关键脚本：`EnemyController.cs`、`EnemyAnimationController.cs`、`EnemyContactDamage.cs`、`EnemyHealth.cs`

10. 背包与道具系统

运行时背包数据采用 `Dictionary<ItemType, int>` 存储，与 ScriptableObject 只读配置完全隔离。`InventoryUI` 统一管理独立背包面板（12 格）、6 格装备栏、快捷栏、箱子存储（16 格）、分页浏览、物品详情展示、丢弃/销毁操作。箱子与背包之间的物品转移基于 `EventSystem` 的拖拽回调实现：`ItemSlotUI` 负责转发 `IBeginDrag / IDrag / IEndDrag`，`InventoryDropArea` 负责接收 `IDropHandler` 事件，最终统一委托给 `InventoryUI.OnDroppedToArea()` 根据槽位归属类型执行“背包→箱子”或“箱子→背包”的数据迁移；若拖拽结束时未落在 UI 区域，则会调用 `TryDropToWorld()` 将物品丢到场景中。拾取掉落物时由 `DroppedItem` 通过碰撞检测触发 `PlayerInventory.AddItem()`，背包变更后通过 C# 事件 `OnInventoryChanged` 通知 UI 刷新，业务逻辑与界面表现完全分离。

> 脚本：`PlayerInventory.cs`、`InventoryUI.cs`、`ItemSlotUI.cs`、`ChestUI.cs`、`DroppedItem.cs`

 11. 面向接口的交互与伤害检测

定义 `IDamageable`（可受伤）、`IInteractable`（可交互）、`IHarvestable`（可采集）三个契约接口，`ResourceNode`、`EnemyHealth`、`Workbench`、`Chest` 等按需实现，玩家侧代码只依赖接口而非具体类型，天然支持多态扩展。`PlayerInteraction` 用 `Physics2D.OverlapCircle` 检测范围内可交互对象，通过 `is IHarvestable` 模式匹配区分采集与普通交互；`PlayerCombat` 在挥击动画关键帧（Animation Event）触发结算：以玩家为顶点、`OverlapCircleAll` 外接圆粗筛后按距离与朝向夹角过滤扇形内目标，对 `IDamageable` 结算伤害（含手持工具加成），一套逻辑同时作用于敌人和可破坏物体。

> 脚本：`IDamageable.cs`、`IInteractable.cs`、`IHarvestable.cs`、`PlayerInteraction.cs`、`PlayerCombat.cs`、`ResourceNode.cs`

12. 任务引导系统

`QuestManager` 订阅 `GameEvents.OnItemPickedUp` 和 `OnBuildingPlaced` 等全局事件，驱动「收集木材→放置营火→存活过夜」三阶段任务推进，每阶段完成后自动发放奖励到背包，并通知 `QuestUI` 更新面板。任务进度随存档持久化，读档后恢复到断点状态。

> 脚本：`QuestManager.cs`、`QuestUI.cs`、`QuestCompleteUI.cs`

 13. 死亡掉落与重生

`PlayerRespawn` 在生命值归零时禁用 `PlayerController`、`PlayerInteraction`、`BuildingSystem` 等组件，切换到 `DeadState`，广播 `GameEvents.OnPlayerDied` 通知 UI 弹出死亡面板。按 R 重生后恢复所有组件、重置属性、回到出生点，广播 `OnPlayerRespawned`，各系统各自响应恢复逻辑。

> 脚本：`PlayerRespawn.cs`、`DeathUI.cs`、`PlayerStats.cs`

14. 采集与资源点

`ResourceNode` 同时实现 `IInteractable` 和 `IHarvestable`，内部维护击打剩余次数 `_currentHitCount`，每次交互扣减，归零后触发掉落并切换枯竭贴图。支持工具需求（`requiresTool`）——未装备指定工具时 `CanHarvest()` 返回 false，交互提示自动切换为「需要 XX 工具」；装备对应工具后读取 `ItemData.harvestHitsPerSwing`，可一次清空多段击打，体现装备对采集效率的提升。可恢复资源通过协程 `WaitForSeconds(regrowTime)` 定时还原贴图与击打次数。掉落物生成优先走 `PoolManager.GetObject()`，池内无可用实例时降级为 `Instantiate`，掉落位置在随机偏移范围内散布。

> 脚本：`ResourceNode.cs`、`IHarvestable.cs`、`DroppedItem.cs`、`PlayerInteraction.cs`

15. 主菜单与场景切换

主菜单通过 `SceneManager.LoadScene()` 同步加载游戏场景。New Game 和 Continue 均调用同一接口，区别在于 Continue 之前先将静态标志位 `SaveManager.ShouldLoadOnSceneStart = true`，游戏场景加载完成后 `SaveManager.Start()` 检测到该 flag 即自动执行 `LoadGame()`，读完存档后立即将 flag 重置为 false 防止重复触发。选用静态字段而非 `DontDestroyOnLoad` 跨场景传值，避免了主菜单对象残留进入游戏场景引发重复实例问题。Continue 按钮的可交互状态由 `SaveManager.HasSaveFileOnDisk()` 静态方法在主菜单 Start 时查本地文件决定，无需 SaveManager 实例。场景加载完成后通过 `[RuntimeInitializeOnLoadMethod(AfterSceneLoad)]` 做一次 Bootstrap，强制同步暂停菜单、背包、工作台等面板的关闭状态，规避执行顺序导致的 UI 初始状态异常。

> 脚本：`MainMenuUI.cs`、`SaveManager.cs`、`PauseMenuSceneBootstrap.cs`

16. 工作台合成系统

`WorkbenchUI` 管理工作台合成面板：左侧配方槽位展示图标与解锁状态，右侧显示配方名称、消耗材料（TMP 富文本标红不足项）、制作按钮。`BuildingRecipeSO` 通过 `isInventoryRecipe` 字段区分两种产出路径——为 true 时合成结果直接加入背包（如石片、火把），为 false 时走世界放置流程（如营火、围墙），同一套配方 SO 同时承载建造与合成两种玩法，新增配方只需新建 SO 资产。工作台范围检测通过 `Workbench.RegisteredInstances` 静态列表遍历，避免每次建造/合成都 `FindObjectsOfType` 全场景扫描。营火的 `Campfire.RegisteredInstances` 同理，供黑暗伤害、任务检测等高频逻辑使用。

> 脚本：`WorkbenchUI.cs`、`BuildingRecipeSO.cs`、`Workbench.cs`、`Campfire.cs`


 演示流程
1. 主菜单：New Game / Continue / Quit，Continue 基于本地存档判定是否可用。
2. 白天采集：按任务提示拾取木材、石头等基础资源，右上角饥饿/生命实时变化。
3. 营火建造：满足配方后放置营火，完成第一个任务节点。
4. 工作台扩展：建造工作台，在合成界面制作石片、火把等进阶物品。
5. 黄昏过渡：时间推进到黄昏，画面变暗，为夜晚风险做铺垫。
6. 夜晚生存：敌人激活追击，远离营火持续掉血，靠近营火安全区完成生存任务。
7. 存档读档：F5 保存进度，F9 或主菜单 Continue 恢复，验证持久化能力。
